**Severity:** HIGH | **Effort:** Large (incremental) | **Priority:** P1 | **Status:** [ ]

### 3. GC Is Non-Generational, Stop-the-World
**Severity:** MEDIUM-HIGH | **Effort:** Incremental | **Priority:** P1 | **Status:** [x] Batch lock done

**Algorithm:** Trial-deletion cycle detector (supplementary to reference counting). Four phases: snapshot, trial-decrement, scan/restore, collect. Per-child lock acquisition during phases 2-3 is a performance concern.

**Next Steps (incremental path):**
1. **Reduce lock granularity** (1 file, low risk): Hold lock for entire phase instead of per-child acquire/release. Snapshot already guards against table mutation.
2. **Add epoch tagging** (1-2 files): Objects surviving N passes get "promoted" (skipped in future passes).
3. **Partial scanning** (1-2 files): Only scan objects tracked since last collection.
4. **Nursery allocator** (4-6 files, high complexity): Long-term goal.

### 4. Runtime Error Recovery — Gap Is Narrower Than Initially Assessed
**Severity:** HIGH | **Effort:** Small to Medium | **Priority:** P0 | **Status:** [ ]

**CORRECTED:** The IL and VM already have full EH infrastructure:
- **IL opcodes:** `eh.push`, `eh.pop`, `eh.entry`, `resume.same`, `resume.next`, `resume.label`, `err.get.kind/code`, `trap.kind` — all fully specified
- **VM:** `VM::prepareTrap()` walks exec stack, finds EH handlers, transfers control to handler blocks. Works correctly.
- **Runtime C:** `rt_trap()` in `rt_io.c:142-150` already checks for thread-local `jmp_buf` recovery point.

**The actual gap:**
1. `rt_trap()` doesn't call `vm_raise()` — falls through to `exit(1)` instead of VM handler dispatch
2. **Native codegen EH is entirely stubbed** — `Lowering.EH.cpp:23-44` all no-ops
3. Runtime C functions call `rt_trap()` directly (23+ files), bypassing VM EH

**Next Steps:**
1. **Wire rt_trap to VM EH** (1-2 files, 5-10 lines, highest impact)
2. **Implement native codegen EH for x86-64** (3-5 files): `setjmp`/`longjmp` mechanism
3. **Implement native codegen EH for AArch64** (3-4 files): Same pattern
4. **Add structured error types** (2-3 files)

### 5. No Debug Info Emission
**Severity:** HIGH | **Effort:** Large | **Priority:** P1 | **Status:** [—]

**Finding:** Zero DWARF/CodeView debug info is emitted anywhere in the codegen pipeline. No `.debug_info`, `.debug_line`, `.debug_aranges` sections. No CodeView sections for Windows.

**Impact:**
- Native Viper binaries cannot be debugged with GDB/LLDB/WinDbg at source level
- Stack traces show symbol names but no line numbers
- Profilers cannot map native addresses back to Viper source code

**Root cause:** IL-to-MIR lowering discards source locations. No location tracking through the pipeline.

**Next Steps:**
1. Add `SourceLoc` field to MIR instructions (both backends) — propagate from IL instructions
2. Emit `.debug_line` (DWARF) section in x86-64/AArch64 AsmEmitters mapping addresses to source lines
3. Emit CodeView `.cv_loc` for Windows
4. Full DWARF variable tracking is a much larger effort (long-term)

**Files:** `src/codegen/x86_64/AsmEmitter.cpp`, `src/codegen/aarch64/AsmEmitter.cpp`, both `MachineIR.hpp`

### 6. No PIE/ASLR Support
**Severity:** MEDIUM | **Effort:** Small-Medium | **Priority:** P1 | **Status:** [x]

**Finding:** No `-fPIC`/`-fPIE` flags, no GOT/PLT indirection. Executables are position-dependent.
- x86-64: RIP-relative addressing for rodata only (strings, floats), not generalized
- AArch64: ADRP + LDR for large immediates, no GOT

**Next Steps:**
1. Add `-fPIE` to linker invocation flags in `LinkerSupport.cpp` (simplest — leverages system assembler)
2. For full PIE: ensure all global references use RIP-relative (x86-64) or ADRP+ADD (AArch64)
3. Test on ASLR-enabled systems (macOS enforces ASLR by default)

**Files:** `src/codegen/common/LinkerSupport.cpp`, `src/codegen/x86_64/AsmEmitter.cpp`

### 18. TLS Certificate Chain Validation — Linux Only Single-Level
**Severity:** HIGH (security) | **Effort:** Medium | **Priority:** P1 | **Status:** [x] Cert expiry checks added

**Finding:** From-scratch TLS 1.3 implementation. ChaCha20-Poly1305 + X25519. Platform-specific cert validation.

**Linux issue** (`rt_tls.c:1541-1630`): Single-level chain validation only — matches end-entity Issuer DER against CA bundle Subject DER. No intermediate cert signature verification, no expiration checks, no CRL/OCSP. Acknowledged in comments: "Full recursive chain validation requires OpenSSL."

**macOS/Windows:** Use platform trust stores (correct).

**Other TLS strengths:**
- Constant-time Finished MAC via `ct_memcmp()` (line 737-743)
- 32KB transcript buffer with overflow trap
- Record size limits (16384 bytes)
- TLS 1.2 downgrade rejected

**Next Steps:**
1. Add intermediate certificate signature verification on Linux (~200 lines)
2. Add certificate expiration/NotBefore/NotAfter checks (~50 lines)
3. Consider fallback to hostname-only verification if no CA bundle found
