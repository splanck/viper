# ARM64 Backend Fix Plan

Date: 2026-03-28

## Scope

This plan covers the confirmed ARM64 backend issues from the March 28, 2026 review pass, plus two additional robustness items that are source-credible and should be hardened while the backend is under active repair.

The priority order is:

1. Correctness bugs that can miscompile or leak at runtime
2. System-asm parity gaps
3. Encoder and allocator hardening
4. Native-pipeline performance cleanup

## Confirmed Issues

### 1. `cbr` same-target lowering is incorrect

Files:
- `src/codegen/aarch64/TerminatorLowering.cpp`

Problem:
- Conditional branches currently emit both successor copy sequences before the branch decision is resolved.
- If both edges target the same block but pass different block arguments, the second store sequence overwrites the first regardless of branch outcome.

Impact:
- Confirmed miscompile on both native and system-asm paths.
- This is the highest-priority functional bug because it produces wrong results in valid IR.

Implementation:
- Add a special-case in ARM64 conditional-branch lowering for `trueLbl == falseLbl`.
- Do not emit both edge-copy bundles unconditionally.
- Preferred approach:
  - evaluate the condition normally;
  - emit control flow so only the taken edge's phi-copy sequence executes;
  - then fall through or branch to the shared target.
- Alternative approach:
  - materialize selected values into temporaries first, then emit one final store bundle.
- Keep the current distinct-successor fast path unchanged.

Validation:
- Add a regression test for `cbr %cond, done(7), done(13)` returning the taken value.
- Add a multi-argument variant to confirm correct handling of multiple block parameters.

### 2. Add/sub immediate legalization is missing or incomplete

Files:
- `src/codegen/aarch64/InstrLowering.cpp`
- `src/codegen/aarch64/fastpaths/FastPaths_Call.cpp`
- `src/codegen/aarch64/binenc/A64BinaryEncoder.cpp`
- Any text-emitter path that prints raw add/sub immediates

Problem:
- Lowering and fast paths can emit `AddRI` and related forms with arbitrary immediates.
- The native encoder assumes the immediate already fits a legal AArch64 add/sub-immediate encoding.
- The system-asm path can also emit invalid text such as large positive immediates that the assembler rejects.

Impact:
- Confirmed native assertion failures for large positive and negative immediates.
- Confirmed system-asm failure for large positive immediates.
- This affects constant `gep` lowering and likely other immediate-producing paths.

Implementation:
- Introduce a shared legalization helper for add/sub immediate forms.
- Helper rules:
  - use `add` for non-negative legal immediates;
  - use `sub` for negative legal immediates;
  - use shifted-12 form when legal and profitable;
  - otherwise materialize the constant in a temp GPR and use register form.
- Update all direct producers of add/sub-immediate MIR to use the helper instead of assuming the immediate fits.
- Keep encoder assertions, but make them post-legalization sanity checks rather than primary validation.

Validation:
- Add tests for:
  - `gep %x, 5000`
  - `gep %x, -8`
  - boundary values around the 12-bit immediate range
  - one shifted-12 legal case if supported by current MIR patterns
- Run each in both native and system-asm modes.

### 3. Stack-local `str` overwrite leaks previous contents

Files:
- `src/codegen/aarch64/InstrLowering.cpp`
- Reference behavior in `src/vm/OpHandlers_Memory.hpp`

Problem:
- The local alloca string-store path retains and stores the new value, but does not load and release the previous slot contents before overwrite.
- The generic pointer-store path already performs release-then-retain.

Impact:
- Confirmed ownership mismatch versus VM semantics.
- Confirmed leak on repeated stores into the same stack slot.

Implementation:
- Change local `str` store lowering to:
  - load old slot value;
  - release old value if needed;
  - retain incoming value;
  - store new value.
- Prefer factoring shared retain/release/store sequencing so local and generic stores do not diverge again.
- Review whether null-handling and call ordering match the generic path exactly.

Validation:
- Add a regression that repeatedly overwrites the same local string slot and verifies release/retain behavior.
- Compare behavior with VM execution semantics.

### 4. FPR-only callee-saves are dropped on the system-asm path

Files:
- `src/codegen/aarch64/AsmEmitter.cpp`
- Reference parity in `src/codegen/aarch64/binenc/A64BinaryEncoder.cpp`

Problem:
- The text emitter recognizes that saved FPRs require a frame, but only enables the frame-plan path when saved GPRs or locals exist.
- A function with only saved FPRs therefore misses the FPR save/restore logic.

Impact:
- Confirmed source bug on the system-asm path.
- Not a native built-in assembler bug, but still a backend correctness issue for system-asm emission.

Implementation:
- Make the text-emitter frame-plan gate include saved FPRs.
- Align the frame-decision logic with the binary encoder to prevent drift.
- Audit prologue and epilogue selection for other plan-vs-default inconsistencies while touching this code.

Validation:
- Add a regression where an FP value lives across a call and requires callee-saved FP preservation in system-asm mode.
- Inspect emitted assembly for save/restore of the appropriate `d8`-`d15` registers.

### 5. Arbitrary FP immediates are emitted incorrectly on the system-asm path

Files:
- `src/codegen/aarch64/AsmEmitter.cpp`
- Reference behavior in `src/codegen/aarch64/binenc/A64BinaryEncoder.cpp`

Problem:
- The text emitter always prints `fmov dN, #imm`, but AArch64 only supports a restricted immediate set for direct FP-immediate forms.
- The native encoder already contains a correct fallback: materialize bits in a GPR, then `fmov d, x`.

Impact:
- Confirmed system-asm failure for `ret 3.14`.
- This is a correctness issue for the explicit text/system-asm path.

Implementation:
- Share or duplicate the native encoder's FP-immediate encodability check.
- Emit direct `fmov #imm` only when the constant is encodable.
- Otherwise:
  - materialize the bit pattern in a GPR;
  - emit `fmov d, x`.
- Prefer one shared utility so the text emitter and binary encoder use the same legality test.

Validation:
- Add a system-asm regression for `ret 3.14`.
- Add one positive test for an encodable FP immediate to ensure the direct form is still used when legal.

## Additional Hardening Items

### 6. SP-relative outgoing-argument stores need explicit large-offset handling

Files:
- `src/codegen/aarch64/binenc/A64BinaryEncoder.cpp`
- `src/codegen/aarch64/InstrLowering.cpp`
- `src/codegen/aarch64/fastpaths/FastPaths_Call.cpp`

Problem:
- SP-relative stack-argument stores appear to encode directly without a visible oversized-offset fallback.
- Large outgoing stack-argument areas could exceed the legal immediate range and misencode.

Impact:
- Not dynamically reproduced yet.
- Source evidence is strong enough to harden while the immediate-legalization work is in progress.

Implementation:
- Add explicit range validation for SP-relative store forms in the encoder.
- For oversized offsets:
  - materialize an address in a temp GPR;
  - use a register-based or temp-base addressing sequence.
- Audit both integer and FP argument-store paths.

Validation:
- Add a stress test with enough stack arguments to force a large outgoing-arg area.

### 7. FPR-pool exhaustion should not silently fall back to the scratch FPR

Files:
- `src/codegen/aarch64/ra/RegPools.cpp`
- `src/codegen/aarch64/ra/Allocator.cpp`

Problem:
- When the FPR pool is empty, the allocator returns the scratch FPR.
- That is risky if callers assume they received a uniquely allocatable physical register.

Impact:
- Not yet reproduced as a miscompile.
- This is a robustness risk under high FP register pressure.

Implementation:
- Review all `takeFPR()` callers and separate:
  - true scratch-register requests;
  - physical-allocation requests that must spill or retry when no register is available.
- Preferred fix:
  - make exhaustion trigger an explicit spill/retry path instead of returning `kScratchFPR`.
- Minimum safeguard:
  - add assertions in allocation contexts where scratch fallback is invalid.

Validation:
- Add a high-pressure FP test with many simultaneous live FP values and spills.

## Performance Cleanup

### 8. Native-only builds should skip text emission

Files:
- `src/codegen/aarch64/CodegenPipeline.cpp`
- `src/codegen/aarch64/CodegenPipeline.hpp`

Problem:
- The pipeline always runs the text emission pass even when only native object output is needed.
- Native mode is the default, so this overhead is paid on the common path.

Impact:
- Confirmed unnecessary work.
- No correctness impact, but this should be cleaned up after the functional bugs are resolved.

Implementation:
- Gate `EmitPass` on whether text assembly is actually required for the chosen output mode or diagnostics.
- Keep `BinaryEmitPass` as the sole required emission step for native object generation.
- Audit whether any existing debug output implicitly depends on `EmitPass`; if so, split those responsibilities cleanly.

Validation:
- Run the existing ARM64 test suite in native mode.
- If there is timing or pass-accounting infrastructure, record a before/after sanity check.

## Execution Order

Recommended implementation order:

1. Fix same-target `cbr` lowering
2. Centralize add/sub immediate legalization
3. Fix local `str` overwrite ownership
4. Fix FPR-only callee-save handling in the text emitter
5. Fix non-encodable FP-immediate text emission
6. Harden SP-relative outgoing-arg store offsets
7. Harden FPR exhaustion behavior in register allocation
8. Remove unnecessary text emission from native-only builds

This order prioritizes confirmed runtime correctness first, then system-asm parity, then encoder and allocator hardening, and finally performance cleanup.

## Refactoring Guidance

- Centralize AArch64 immediate legalization in shared helpers rather than repeating policy across lowering, text emission, and binary encoding.
- Unify frame-decision logic between the text emitter and binary encoder so they cannot drift on save/restore behavior.
- Factor refcounted store lowering so local-stack and generic-memory stores share one ownership implementation.
- Keep encoder assertions even after legalization is fixed so future regressions fail loudly.

## Test Plan

Add targeted regressions for:

- same-target `cbr` with distinct block arguments
- large positive and negative constant `gep` offsets in native mode
- large positive constant `gep` offset in system-asm mode
- repeated stack-local `str` overwrite with refcount verification
- FP callee-save preservation on the system-asm path
- non-encodable FP immediate return on the system-asm path
- large outgoing stack-argument areas
- FP register-pressure stress for allocator behavior

## Delivery Strategy

Recommended patch sequence:

1. Land confirmed correctness fixes with regressions
2. Land text/system-asm parity fixes with regressions
3. Land encoder and allocator hardening
4. Land native-pipeline performance cleanup

Each patch should include:

- the minimal functional change for that issue;
- one targeted regression test;
- any shared refactor needed to prevent recurrence.
