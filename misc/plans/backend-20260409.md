# Backend Codegen, Assembler, Linker — Deep Audit Findings

## Context

Deep technical audit of `src/codegen/aarch64/`, `src/codegen/x86_64/`, `src/codegen/common/linker/`, and `src/codegen/common/objfile/` to identify bugs, correctness risks, and optimization opportunities. Each finding was verified by directly reading the quoted code. Findings that looked plausible in the initial sweep but did not survive verification are listed at the bottom as "Rejected on Verification" for honesty.

## Verification Methodology

Each candidate finding from the initial sweep was checked by:
1. Reading the actual cited file at the cited line numbers
2. Tracing whether the claimed scenario is reachable from current codegen output
3. Classifying confidence as HIGH (verified bug or clear code smell), MEDIUM (real issue, narrow/latent trigger), LOW (theoretical, rejected)

## Top 10 Verified Findings (Ranked by Severity)

### 1. Debug `fprintf` left in production x86_64 backend (HIGH / bug, trivial fix)

**File**: `src/codegen/x86_64/Backend.cpp:425`

```cpp
for (std::size_t i = 0; i < mir.size(); ++i) {
    fprintf(stderr, "[emit] %zu: %s\n", i, mir[i].name.c_str()); fflush(stderr);
```

Unconditional debug print in `emitMIRToBinary`. Every native x86_64 compilation writes one line per function to stderr. This pollutes `viper build` output, slows compilation, and will show up in CI logs and end-user installations.

**Fix**: Delete the line, or gate behind `opt.verbose`/`VIPER_DEBUG_CODEGEN`.
**Confidence**: HIGH — verified by direct read.

### 2. ICF address-taken detection skips executable sections (MEDIUM / latent bug)

**File**: `src/codegen/common/linker/ICF.cpp:155-188`

```cpp
for (size_t si = 1; si < obj.sections.size(); ++si) {
    const auto &sec = obj.sections[si];
    // Only scan data/rodata sections (non-executable, allocatable).
    if (sec.executable || sec.data.empty())
        continue;
    for (const auto &rel : sec.relocs) {
        ...
        if (obj.sections[targetSym.sectionIndex].executable)
            addressTaken.insert(targetSym.name);
```

ICF collects address-taken function names by scanning relocations in non-executable sections. It deliberately skips `.text` sections. A jump table embedded in `.text` with `Abs64` relocations pointing to sibling functions would NOT mark those sibling functions as address-taken. ICF could then fold two semantically distinct functions, breaking runtime behavior when one is invoked via the jump table.

**Current exposure**: Low. Viper's codegen does not emit in-text jump tables today (verified with `grep -r "jump_table\|jumptable\|jtbl\|JumpTable" src/codegen` — no hits). This is a latent bug that only matters if jump-table codegen is added later.

**Fix**: Also scan executable sections when building `addressTaken`, or add a comment documenting the assumption that Viper never emits in-text jump tables.
**Confidence**: HIGH (code behavior verified) / MEDIUM (active exploitability depends on future codegen).

### 3. ELF dynamic section missing DT_DEBUG and DT_GNU_HASH (MEDIUM / opportunity)

**File**: `src/codegen/common/linker/ElfExeWriter.cpp:62-73`

```cpp
// Dynamic tags.
static constexpr int64_t DT_NULL = 0;
static constexpr int64_t DT_NEEDED = 1;
static constexpr int64_t DT_HASH = 4;
static constexpr int64_t DT_STRTAB = 5;
static constexpr int64_t DT_SYMTAB = 6;
static constexpr int64_t DT_RELA = 7;
static constexpr int64_t DT_RELASZ = 8;
static constexpr int64_t DT_RELAENT = 9;
static constexpr int64_t DT_STRSZ = 10;
static constexpr int64_t DT_SYMENT = 11;
static constexpr int64_t DT_TEXTREL = 22;
```

The dynamic tag list is missing two entries that matter for real-world ELF usage:

- **DT_DEBUG (21)**: required for `gdb` to attach via `dlopen`'d libraries. Without it, gdb can still debug the main binary, but cannot see dynamic loader state. This affects developer experience, not correctness.
- **DT_GNU_HASH (0x6ffffef5)**: GNU hash section for fast symbol lookup. The current SYSV `DT_HASH` works but is slower. Modern glibc prefers DT_GNU_HASH.

**Note on the subagent's DT_PLTGOT/DT_JMPREL claim**: That claim was wrong. Viper's ELF dynamic-link strategy uses GOT-based eager binding via `generateDynStubsX8664()` — `jmpq *__got_sym(%rip)` stubs with GOT slots populated by R_X86_64_GLOB_DAT at load time. There is no PLT. DT_PLTGOT/DT_JMPREL/DT_PLTRELSZ are intentionally absent.

**Fix**: Add DT_DEBUG (easy, one relocatable slot). DT_GNU_HASH is bigger work (hash table construction) and optional.
**Confidence**: HIGH.

### 4. AArch64 cross-block reload defaults FPR temps to GPR when class missing (MEDIUM / fragile)

**File**: `src/codegen/aarch64/LowerILToMIR.cpp:396-410`

```cpp
// Check register class for this temp
auto clsIt = tempRegClass.find(op.id);
const RegClass cls =
    (clsIt != tempRegClass.end()) ? clsIt->second : RegClass::GPR;
if (cls == RegClass::FPR) {
    bbOutFn().instrs.push_back(
        MInstr{MOpcode::LdrFprFpImm, ...});
} else {
    bbOutFn().instrs.push_back(
        MInstr{MOpcode::LdrRegFpImm, ...});
}
```

When a cross-block temp is reloaded, the register class is looked up in `tempRegClass`. If the map doesn't contain an entry, the code silently defaults to GPR. If an FPR temp is ever reloaded without having its class registered, a `Ldr` (GPR load) will be emitted where `LdrFpr` should have been — causing silent corruption of the floating-point value.

**Current exposure**: All known FPR-producing lowering paths populate `tempRegClass` when emitting the definition. But there is no assertion to catch the case where it doesn't, and the default fails silently rather than loudly.

**Fix**: Replace the default with `assert(clsIt != tempRegClass.end() && "cross-block temp has no register class")` or throw a hard error. This surfaces the bug at codegen time instead of producing corrupted float values.
**Confidence**: HIGH (code verified) / MEDIUM (trigger requires a lowering path that forgets to set class).

### 5. Magic number `60000` in vreg ID range check (MEDIUM / code smell)

**File**: `src/codegen/aarch64/LowerILToMIR.cpp:390-391`

```cpp
if (tempVReg.find(op.id) == tempVReg.end() ||
    tempVReg[op.id] < 60000) // Use high range for reloaded values
```

The number `60000` is hardcoded with a cryptic comment. Checking `src/codegen/aarch64/ra/` and the vreg ID partitioning documented elsewhere, there are named constants `kFirstVirtualRegId`, `kPhiVRegStart=40000`, `kCrossBlockSpillKeyStart=50000` for similar partitions. This file uses a separate, unnamed constant that doesn't match any of those.

**Impact**: If `kCrossBlockSpillKeyStart` is bumped to 60000+ somewhere else, this check silently breaks. Maintenance hazard.

**Fix**: Promote `60000` to a named constant (e.g. `kReloadedVRegStart`) in a shared header, or reuse one of the existing constants if semantics align.
**Confidence**: HIGH.

### 6. LowerOvf operand count check is too loose (LOW / defensive hardening)

**File**: `src/codegen/x86_64/LowerOvf.cpp:124-126`

```cpp
if (instr.operands.size() < 2U) {
    continue;
}
```

Overflow pseudo-ops `ADDOvfrr`, `SUBOvfrr`, `IMULOvfrr` are three-operand instructions (dst, lhs, rhs). The check passes any instruction with ≥ 2 operands, then clones whatever operands exist into the real ADD/SUB/IMUL instruction. A malformed 2-operand input would silently produce a malformed 2-operand arithmetic instruction, which the binary encoder would eventually reject.

**Current exposure**: None. No lowering path produces 2-operand Ovf pseudos. This is a hardening issue only.

**Fix**: Change `< 2U` to `< 3U`. One-character fix.
**Confidence**: HIGH (code verified) / LOW (unreachable from current MIR).

### 7. BranchTrampoline: unclamped boundary stored in placements (MEDIUM / latent)

**File**: `src/codegen/common/linker/BranchTrampoline.cpp:291-315`

```cpp
for (const auto &[boundary, entries] : islands) {
    const size_t clampedBoundary = std::min(boundary, originalText.size());
    newText.insert(newText.end(),
                   originalText.begin() + static_cast<std::ptrdiff_t>(cursor),
                   originalText.begin() + static_cast<std::ptrdiff_t>(clampedBoundary));
    ...
    placements.push_back({boundary, entries.size() * kTrampolineSize});
    cursor = clampedBoundary;
}
...
for (auto &chunk : textSec.chunks) {
    size_t shift = 0;
    for (const auto &placement : placements) {
        if (chunk.outputOffset >= placement.boundary)
            shift += placement.size;
    }
    chunk.outputOffset += shift;
}
```

The insertion point is `clampedBoundary` but the record pushes `boundary` (unclamped). Chunk shift logic compares `chunk.outputOffset >= placement.boundary` using the unclamped value. If `boundary > originalText.size()`, chunks in the range `[clampedBoundary, boundary)` would miss the shift even though the island was physically inserted before them.

**Current exposure**: The `boundary` values come from `chooseReachableBoundary()` which picks from `collectChunkBoundaries()` — all chunk offsets are inside the text section. In practice the `std::min` clamp never fires. The defensive clamp is correct; the bug would only manifest if a caller passes a pathological boundary.

**Fix**: Use `clampedBoundary` when constructing the placement. One-line fix. Removes a latent hazard even though it's not currently reachable.
**Confidence**: HIGH (logic verified) / MEDIUM (only triggerable if the clamp ever fires).

### 8. DeadStripPass synthetic-object detection uses name prefix `<` (LOW-MEDIUM / fragile)

**File**: `src/codegen/common/linker/DeadStripPass.cpp:89-96`

```cpp
for (size_t oi = userObjCount; oi < allObjects.size(); ++oi) {
    const auto &obj = allObjects[oi];
    if (obj.name.size() > 0 && obj.name[0] == '<') {
        // Synthetic objects like <dyld-stubs>, <objc-stubs>
        for (size_t si = 1; si < obj.sections.size(); ++si)
            markLive(oi, si);
    }
}
```

Root marking for linker-synthesized objects relies on the first character of `obj.name` being `<`. In current code, all synthetic objects are generated by the linker itself in `DynStubGen.cpp` (`<elf64-dyn-stubs>`, `<aarch64-dyn-stubs>`, etc.), so names are controlled. But a user archive with `<name>.o` would be incorrectly treated as a linker-internal synthetic object, skipping its dead-strip analysis.

**Current exposure**: Low — Unix archive conventions discourage `<` in filenames, and the code applies only to entries with `oi >= userObjCount` (archive extracts). But there's no enforcement.

**Fix**: Add an explicit `isSynthetic` flag on `ObjFile` and set it in `DynStubGen` instead of relying on name prefix. Cleaner, harder to break accidentally.
**Confidence**: HIGH (behavior verified) / LOW (not currently exploitable).

### 10. AArch64 branch relaxation iteration count uses over-conservative bound (LOW / cosmetic)

**File**: `src/codegen/aarch64/binenc/A64BinaryEncoder.cpp:141-181`

```cpp
size_t relaxCandidates = 0;
for (const auto &bb : fn.blocks) {
    for (const auto &mi : bb.instrs) {
        if (mi.opc == MOpcode::BCond || mi.opc == MOpcode::Cbz || mi.opc == MOpcode::Cbnz)
            ++relaxCandidates;
    }
}

const size_t maxIterations = std::max<size_t>(2, relaxCandidates + 1);
```

On AArch64, every instruction is exactly 4 bytes — including `BCond`, `Cbz`, `Cbnz`. None of them have a "short" encoding that could change size during relaxation. The iteration loop will converge in at most 2 passes (first pass fills offsets, second pass confirms stability). Bounding iterations by `relaxCandidates + 1` is correct but misleading — the variable name implies there's actual relaxation happening, and a future reader may copy this pattern to a context where it does matter (x86_64, where short Jcc does exist).

**Fix**: Rename variable and cap iterations at a small constant (e.g. `kMaxLabelPasses = 4`) with a comment explaining that AArch64 instructions are fixed-size. Pure code clarity; no functional change.
**Confidence**: HIGH.

## Rejected on Verification (Transparency)

These findings from the initial audit sweep did NOT survive direct code verification and should NOT be acted on:

- **"ELF dynamic missing DT_PLTGOT/DT_JMPREL causes linking to fail"** — Viper uses GOT-based eager binding via R_X86_64_GLOB_DAT, not PLT lazy binding. These tags are intentionally absent.
- **"AArch64 spillVictim() leaves state incoherent when vreg has no more uses"** — When `nextUseDist == UINT_MAX && !liveOut`, nothing will read the state again. The desync is harmless because there's no subsequent reader.
- **"StringDedup section compaction corrupts cross-object symbol offsets"** — Compaction happens AFTER Step 2 (rename to synthetic names), and only within a single `(objIdx, secIdx)` group. The cross-object corruption path described doesn't exist.
- **"MachO __objc_classname missing S_ATTR_NO_DEAD_STRIP causes selector loss"** — These cstring sections are referenced from `__objc_selrefs`/`__objc_classlist` which DO carry NO_DEAD_STRIP. DeadStripPass follows relocations from live sections, so cstrings get marked live transitively. Lower severity than claimed.
- **"Trampoline symbol counter could wrap to SIZE_MAX"** — The counter is inside `if (inserted)` and `trampolines.size() - 1` is only evaluated after an insert, so size is always ≥ 1.
- **"X64BinaryEncoder branch relaxation iteration bound could cause performance issue"** — The convergence check terminates early in normal cases. Not an observable problem.
