## Confirmed Issues

### 1. x86_64 backend emits unconditional debug logs

**Verdict**: Actual issue, now fixed

This is still present, and it is worse than the original note implied. The x86_64 backend currently prints unconditionally in multiple phases:

- `src/codegen/x86_64/Backend.cpp` during legalization (`[leg]`)
- `src/codegen/x86_64/Backend.cpp` during allocation (`[alloc]`)
- `src/codegen/x86_64/Backend.cpp` during peepholes (`[peep]`)
- `src/codegen/x86_64/Backend.cpp` during binary emission (`[emit]`)

So the original finding was directionally correct, but under-reported the scope. This is user-visible stderr noise in normal builds and test runs, not just a developer-only trace.

### 5. The `60000` check in AArch64 cross-block reloads is logically broken

**Verdict**: Actual issue, now fixed

The original note called this a magic-number smell. Re-review shows the real problem is stronger: the predicate does not implement what the comment says.

In `src/codegen/aarch64/LowerILToMIR.cpp`, cross-block reloads are guarded by:

```cpp
if (tempVReg.find(op.id) == tempVReg.end() ||
    tempVReg[op.id] < 60000)
```

The comment says "Only reload if we haven't already in this block", but current vreg allocation contradicts that:

- `allocateNextVReg()` allocates ordinary vregs below `kPhiVRegStart`
- `kPhiVRegStart` is `40000` in `src/codegen/aarch64/LoweringContext.hpp`
- every `tempVReg[op.id]` assigned by current lowering is therefore `< 60000`

That makes the second half of the condition effectively always true for ordinary mappings. Result: if the same cross-block temp is referenced multiple times while scanning a block's operands, the lowering code can emit multiple reloads for the same temp and keep overwriting `tempVReg[op.id]` with the newest vreg.

This does not appear to corrupt correctness, but it does create redundant loads and unnecessary vregs. The comment and the implementation are currently out of sync.

## Real But Not Current Bugs

### 2. ICF address-taken scan ignores executable sections

**Verdict**: Latent only

The code still only scans non-executable sections when computing `addressTaken`. That assumption would become unsafe if the backend started emitting function-pointer tables or jump tables into `.text`.

I did not find evidence that the current codegen does that. Current switch lowering is compare-and-branch based rather than jump-table based. So this is not a present bug in the current backend; it is a future-proofing hole.

### 3. ELF writer omits `DT_DEBUG` and `DT_GNU_HASH`

**Verdict**: Real omission, but not a correctness bug

The omission is real. `buildDynamicInfo()` still emits `DT_HASH`, `DT_STRTAB`, `DT_SYMTAB`, `DT_RELA`, `DT_RELASZ`, `DT_RELAENT`, `DT_STRSZ`, `DT_SYMENT`, and optionally `DT_TEXTREL`, but not `DT_DEBUG` or `DT_GNU_HASH`.

That matters for:

- debugger integration (`DT_DEBUG`)
- faster dynamic symbol lookup (`DT_GNU_HASH`)

It does not make the generated executable incorrect under the current dynamic-link strategy. I would keep this on a feature backlog, not in the "verified bugs" list.

### 4. Missing `tempRegClass` entry would mis-reload an FPR temp

**Verdict**: Hardening only

The fallback-to-GPR behavior is real in the code, but I did not find a current lowering path that produces a cross-block FPR temp without populating `tempRegClass`.

Current AArch64 lowering records FPR class information for:

- function and block parameters
- floating-point op results
- floating-point call results
- relevant conversion paths

So the original concern is valid as a defensive assertion request, but I could not validate it as a present bug.

### 8. DeadStrip synthetic-object detection via `obj.name[0] == '<'`

**Verdict**: Latent robustness issue

This heuristic is still present. It is brittle because `deadStrip()` treats all post-`userObjCount` objects whose names start with `<` as synthetic roots.

That said:

- true linker-generated objects do use this naming convention
- the dangerous collision requires an archive-extracted member whose diagnostic name starts with `<`
- `userObjCount` is `initialObjects.size()`, so only archive extracts and linker-added objects are in the affected range

I would not call this a current bug, but it is a real cleanup candidate if the linker code keeps growing.

## False Alarms

### 6. `LowerOvf` operand count check

**Verdict**: False alarm

The original finding assumed the overflow pseudos were 3-operand instructions. In the current x86 lowering pipeline they are not.

`emitBinary()` first moves the left-hand side into the destination register, then emits the arithmetic pseudo using x86-style two-operand form:

- destination already holds the lhs
- instruction carries destination and rhs

So `if (instr.operands.size() < 2U)` is consistent with current MIR shape. The stale part is the comment in `MachineIR.hpp`, not the guard in `LowerOvf.cpp`.

### 7. BranchTrampoline stores `boundary` instead of `clampedBoundary`

**Verdict**: False alarm for current code

The mismatch exists textually, but the supposedly dangerous path is not reachable with current inputs.

`boundary` comes from `chooseReachableBoundary()`, which chooses from `collectChunkBoundaries(textSec)`. Those boundaries are derived from chunk offsets and chunk ends inside the current text section, so they are already within `originalText.size()`. That means the defensive `std::min(boundary, originalText.size())` clamp should never change the value in normal execution.

This is harmless cleanup, not a current bug.

### 9. Signed division strength reduction is disabled

**Verdict**: False alarm

The code explicitly says the old non-power-of-two signed division transform was wrong and intentionally keeps `SDIV` until a proven-correct replacement exists. That is the right safety choice, not a defect.

### 10. AArch64 branch-relaxation iteration bound

**Verdict**: False alarm

The bound is over-explained and not especially elegant, but AArch64 instructions are fixed-width and the loop exits early on convergence anyway. I do not see a functional or measurable correctness problem here.

## Rejected Items Re-Checked

These were already rejected in the earlier note. Re-review did not change that verdict.

- Missing `DT_PLTGOT` / `DT_JMPREL`: still false; the Linux x86_64 path uses GOT-based eager binding stubs, not a PLT.
- AArch64 `spillVictim()` no-more-uses case: still false; the state is intentionally dropped when nothing can observe it again.
- `StringDedup` cross-object compaction corruption: still false; rename-to-synthetic happens before compaction, and compaction is section-local.
- Mach-O `__objc_classname` missing `S_ATTR_NO_DEAD_STRIP`: still false; liveness flows from the live ObjC reference sections.
- Trampoline symbol counter wraparound: still false; `trampolines.size() - 1` is used only on successful insertion.
- x86_64 branch-relaxation iteration count as a performance bug: still false; the loop converges early in normal cases.

## Recommended Action Order

The two present-day fixes from this review are now implemented:

1. Removed the unconditional x86_64 backend logging
2. Replaced the broken AArch64 `60000` reload heuristic with explicit per-block reload tracking

Remaining optional follow-up work:

1. Add an assertion for missing `tempRegClass` on cross-block reload/spill paths
2. Treat `DT_DEBUG` and `DT_GNU_HASH` as linker feature work, not bug work
3. Leave the rest off the immediate bug list
