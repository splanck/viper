# x86_64 Backend And Runtime Audit Plan

Date: 2026-03-28

## Scope

Fresh audit of `src/codegen/x86_64` with emphasis on:

- ABI correctness on SysV and Win64
- register allocation and liveness correctness
- frame lowering and stack probing
- peephole safety after regalloc
- gaps likely to matter before Linux and Windows bring-up

## Findings

### 1. High: liveness analysis writes `preds_` before it is sized

Files:

- `src/codegen/x86_64/ra/Liveness.cpp:41`
- `src/codegen/x86_64/ra/Liveness.cpp:64`

Problem:

- `run()` sizes `succs_`, `gen_`, and `kill_`, but not `preds_`.
- `buildCFG()` writes to `preds_[...]` while `preds_` is still empty.
- `preds_` is then recomputed from `succs_` after the damage is already done.

Impact:

- Undefined behavior in liveness construction.
- Regalloc stability depends on allocator luck rather than valid containers.

Recommended fix:

- Stop mutating `preds_` inside `buildCFG()`, or size `preds_` before the call.
- Prefer a single source of truth: build only `succs_` in `buildCFG()` and derive `preds_` once with `buildPredecessors(succs_)`.

### 2. High: SysV call-site stack alignment is wrong when stack args are present

Files:

- `src/codegen/x86_64/CallLowering.cpp:176`
- `src/codegen/x86_64/FrameLowering.cpp:267`
- `src/tests/codegen/x86_64/test_abi_probe.cpp:107`

Problem:

- `CallLowering.cpp` subtracts dynamic `padBytes` from `%rsp` before the call.
- `FrameLowering.cpp` already folds `frame.outgoingArgArea` into `frame.frameSize`, aligned to 16.
- With reserved outgoing area, the extra runtime subtraction double-adjusts the stack and flips SysV alignment for odd stack-arg counts.

Impact:

- ABI violation at real call boundaries.
- Likely latent breakage in libc calls, varargs, and platform-specific code that assumes proper stack alignment.

Recommended fix:

- Remove the SysV dynamic `padBytes` path when using frame-resident outgoing arg space.
- Recompute `test_abi_probe.cpp` so it stops asserting the current bad `addq $-8, %rsp` behavior.
- Add focused tests for 1, 3, and 5 stack-argument SysV calls.

### 3. High: Unix/macOS large-frame stack probing overshoots the real target

Files:

- `src/codegen/x86_64/FrameLowering.cpp:392`
- `src/tests/unit/codegen/test_codegen_review_x64_frame.cpp:63`

Problem:

- For `frame.frameSize > kPageSize`, the prologue subtracts one page per probe, touches memory, then resets `%rsp` to `%rbp` and subtracts the full frame again.
- A non-page-multiple frame like 4097 bytes probes to 8192 bytes below `%rbp`, not 4097.

Impact:

- Can touch a guard page the real frame would never reach.
- Turns large but valid frames into avoidable crashes.

Recommended fix:

- Probe toward the exact final target:
  - subtract/touch full pages while `remaining > kPageSize`
  - subtract the final tail once
- Strengthen the existing test so it validates final probed depth, not just probe count.

### 4. High: FP argument registers are not reserved during call setup

Files:

- `src/codegen/x86_64/ra/Allocator.cpp:399`
- `src/codegen/x86_64/ra/Allocator.cpp:455`
- `src/codegen/x86_64/ra/Allocator.cpp:981`
- `src/codegen/x86_64/ra/Allocator.cpp:995`

Problem:

- The allocator reserves only GPR argument registers during pre-call setup.
- `isArgumentRegister()` checks only `intArgOrder`.
- `reserveForCall()` and `releaseCallReserved()` operate only on the GPR pool.
- Later spill reloads can reuse XMM argument registers and clobber already-marshalled FP arguments before the `CALL`.

Impact:

- Real miscompile risk for 2+ FP arguments, especially with spills or high pressure near calls.
- Affects SysV and Win64; Win64 is worse because only four FP arg registers exist.

Recommended fix:

- Make reservation ABI-aware across both register classes.
- Reserve XMM0-XMM7 for SysV and XMM0-XMM3 for Win64 during call setup.
- Replace the single `reservedForCall_` vector with per-class reserved sets or a unified typed structure.

### 5. High: peephole DCE models call FP uses incorrectly

Files:

- `src/codegen/x86_64/peephole/DCE.cpp:237`
- `src/codegen/x86_64/Backend.cpp:159`

Problem:

- DCE marks only `XMM0` and `XMM1` live across `CALL`.
- On SysV, up to `XMM7` may hold FP arguments.
- On Win64, up to `XMM3` may hold FP arguments.

Impact:

- Post-regalloc peephole can delete setup for the 3rd+ FP argument.
- This is a codegen correctness bug, not just a missed optimization.

Recommended fix:

- Make call-use modeling ABI-aware.
- Mark all possible FP argument registers live for the active target ABI.
- Add a regression with at least 3 FP register args on SysV and 4 on Win64.

### 6. Low: `SETcc` def/use modeling in x86 DCE is inconsistent

Files:

- `src/codegen/x86_64/peephole/DCE.cpp:86`
- `src/codegen/x86_64/ra/Allocator.cpp:837`
- `src/codegen/x86_64/ra/Liveness.cpp:138`

Problem:

- The backend’s allocator and liveness treat `SETcc` operand 1 as the destination.
- `DCE.cpp` still includes `SETcc` in the generic “operand 0 defines the register” list.
- In practice this returns `nullopt` because operand 0 is the condition-code immediate, so DCE never models the real destination def.

Impact:

- Mostly a missed optimization.
- Can preserve dead `SETcc` chains and keep unnecessary flag-producing instructions alive.

Recommended fix:

- Teach `getDefReg()` to special-case `SETcc` operand 1 explicitly.
- Add a unit test where a dead `SETcc` becomes removable only when def/use is modeled correctly.

### 7. Medium: cross-block regalloc still force-spills every live-out value

Files:

- `src/codegen/x86_64/ra/Allocator.cpp:108`
- `src/codegen/x86_64/ra/Allocator.cpp:670`

Problem:

- `run()` pre-marks every vreg that appears in any `liveOut` set as needing a spill slot.
- `releaseActiveForBlock()` spills every live-out value at the end of each block, then clears all active sets.
- That is safe, but it abandons any chance to carry values in registers across edges.

Impact:

- Extra spill/reload traffic on branchy code.
- Larger frames and more memory pressure.
- Likely to matter in control-heavy BASIC/ZIA output.

Recommended fix:

- Keep values live across straight-line/single-successor edges where possible.
- At minimum, avoid unconditional spill when the successor has a single predecessor and register assignment is compatible.
- Longer term, move toward edge-aware RA or block-entry reconciliation rather than “spill all live-outs”.

## Runtime Audit Addendum

Scope note:

- This addendum covers exported/runtime-callable Windows/Linux issues in `src/runtime`.
- It intentionally excludes build-flag stubs such as "graphics not compiled in" paths and unsupported-platform fallbacks that are not specific to Windows/Linux parity.

### Runtime Findings

#### R1. High: Windows filesystem and process APIs are still ANSI-only and not full UTF-8 / long-path implementations

Files:

- `src/runtime/io/rt_file_path.c`
- `src/runtime/io/rt_dir.c`
- `src/runtime/io/rt_tempfile.c`
- `src/runtime/io/rt_binfile.c`
- `src/runtime/io/rt_linereader.c`
- `src/runtime/io/rt_linewriter.c`
- `src/runtime/io/rt_savedata.c`
- `src/runtime/io/rt_archive.c`
- `src/runtime/io/rt_watcher.c`
- `src/runtime/network/rt_network_http.c`
- `src/runtime/system/rt_exec.c`

Problem:

- The Windows path layer still hands raw `char *` buffers through `rt_file_path_from_vstr()`.
- Core runtime code then routes those bytes into ANSI CRT or Win32 APIs such as `fopen`, `CreateFileA`, `FindFirstFileA`, `GetTempPathA`, `_mkdir`, `_chdir`, and `CreateProcessA`.
- `rt_dir.c` already documents that the current `FindFirstFileA` / `FindNextFileA` implementation is limited to `MAX_PATH`.

Impact:

- Windows runtime APIs are not full implementations for UTF-8 paths, non-ASCII filenames, or extended-length paths.
- File I/O, temp files, directory walking, archives, watcher startup, HTTP downloads to file, and `Exec` can all fail on valid Windows paths.

Recommended fix:

- Introduce a shared UTF-8 <-> UTF-16 path/command conversion layer for Windows.
- Migrate filesystem/process call sites to wide APIs:
  - `CreateFileW`
  - `FindFirstFileW` / `FindNextFileW`
  - `_wfopen`
  - `_wmkdir`
  - `_wchdir`
  - `CreateProcessW`
- Add `\\?\\` extended-path handling where appropriate so long paths work consistently.

#### R2. High: Windows watcher file-target support is incomplete and can watch the wrong directory

Files:

- `src/runtime/io/rt_watcher.c`
- `src/il/runtime/runtime.def`

Problem:

- `Viper.IO.Watcher` is an exported runtime class.
- On Windows, `rt_watcher_start()` uses `const char *watch_dir = w->is_directory ? cpath : ".";`.
- For file watches this watches the current working directory instead of the file's parent directory, and there is no filename filter on the resulting `ReadDirectoryChangesW` events.

Impact:

- File-target watchers on Windows are not a full implementation of the public API.
- A watcher can miss changes to the requested file or report unrelated events from the wrong directory.

Recommended fix:

- For file targets, resolve the parent directory and basename up front.
- Open the parent directory handle, then filter queued events to the watched basename.
- Add native Windows tests for:
  - watching a directory
  - watching a single file in a non-CWD directory
  - rename/delete/modify event filtering

#### R3. High: Linux TLS chain validation is only best-effort and not a full trust-store implementation

Files:

- `src/runtime/network/rt_tls_verify.c`
- `src/il/runtime/runtime.def`

Problem:

- `Viper.Crypto.Tls.Connect*` is exported as a public runtime API.
- The Linux chain verifier explicitly documents itself as "best-effort".
- It hardcodes a short CA-bundle file list, scans PEM files manually, and accepts a chain only when the end-entity issuer matches a CA subject found in the bundle.
- The code comments also admit that full recursive chain validation requires OpenSSL.

Impact:

- Linux TLS validation is materially weaker and less complete than the Windows/macOS paths.
- Valid chains with intermediates can fail, and trust-store behavior depends on a narrow set of bundle file locations.

Recommended fix:

- Replace the manual issuer-subject scan with full chain validation on Linux.
- Prefer an OpenSSL-backed path using `X509_STORE` / `X509_STORE_CTX`, or an equivalent shared verifier abstraction.
- Support both CA bundle files and CA directories used by major distros.
- Keep hostname validation separate, but ensure chain validation itself is complete.

#### R4. High: CertificateVerify hashing is hard-coded to SHA-256, so Windows/Linux signature-scheme coverage is incomplete

Files:

- `src/runtime/network/rt_tls_verify.c`

Problem:

- `build_cert_verify_content()` always hashes the CertificateVerify input with SHA-256.
- Windows and Linux both advertise or branch on non-SHA-256 signature schemes such as:
  - `0x0503` ECDSA P-384 + SHA-384
  - `0x0805` RSA-PSS + SHA-384
  - `0x0806` RSA-PSS + SHA-512
- The Linux verifier then hard-codes `hash_len = 32`, with an explicit TODO for SHA-384/SHA-512 support.

Impact:

- Windows/Linux CertificateVerify handling is not complete for the full set of schemes the code claims to recognize.
- Handshakes using SHA-384 or SHA-512-based CertificateVerify signatures can fail or be validated against the wrong digest shape.

Recommended fix:

- Replace the helper with a signature-scheme-aware digest builder.
- Compute the CertificateVerify digest with the hash mandated by the signature scheme:
  - SHA-256
  - SHA-384
  - SHA-512
- Ensure each verifier path uses the correct digest length and algorithm-specific input.

#### R5. High: Windows CertificateVerify public-key verification is incomplete and rejects some key types/schemes

Files:

- `src/runtime/network/rt_tls_verify.c`

Problem:

- The Windows verifier falls back from `CryptAcquireCertificatePrivateKey()` to public-key import, even though server-certificate verification should be a public-key path from the start.
- The function can still fail with:
  - `TLS: CertificateVerify: unsupported scheme (Windows)`
  - `TLS: CertVerify: unsupported key type on Windows`
- That leaves Windows TLS verification incomplete for some advertised combinations.

Impact:

- TLS on Windows is not a full implementation of the exported `Viper.Crypto.Tls` API for the full scheme/key set currently recognized by the handshake parser.
- Linux and Windows can diverge depending on server certificate type and CertificateVerify algorithm.

Recommended fix:

- Rework Windows verification around public-key verification only.
- Use a proper CNG/BCrypt/NCrypt-backed verifier for:
  - ECDSA P-256
  - ECDSA P-384
  - RSA-PSS SHA-256
  - RSA-PSS SHA-384
  - RSA-PSS SHA-512
- Do not rely on acquiring a private key from the peer certificate.
