# Sidescroller Demo Deep Dive: Runtime Gap Analysis

## Context

The sidescroller demo (`examples/games/sidescroller/`) is a ~3000 LOC, 15-file Zia game with impressive scope: 2 levels, 4 enemy types, 5 power-ups, particles, parallax, boss fights, full menus, and 12 sound effects. However, it **reimplements almost everything from scratch** despite the Viper runtime having ready-made systems for most of its needs.

---

## THE CRITICAL FINDING: Unused Runtime Systems

The Viper runtime already exposes these game systems to Zia — **none of which the sidescroller uses:**

| Runtime System | Zia API Path | What Demo Does Instead |
|---|---|---|
| **Physics2D.World + Body** | `Viper.Game.Physics2D.*` | 200+ LOC manual gravity, velocity, AABB collision in `physics.zia` |
| **ParticleEmitter** | `Viper.Game.ParticleEmitter.*` | 150+ LOC custom particle pool in `particles.zia` |
| **Sprite** (w/ animation, flip, collision) | `Viper.Graphics.Sprite.*` | Parallel `List[Pixels]` arrays + manual frame counters in `sprites.zia` (864 LOC!) |
| **SpriteAnimation** | `Viper.Game.SpriteAnimation.*` | Manual `animTimer % frameCount` everywhere |
| **Tilemap** (w/ collision, viewport culling) | `Viper.Graphics.Tilemap.*` | `Grid2D` + 100+ LOC manual tile rendering loop in `renderer.zia` |
| **Camera** (follow, bounds, world↔screen) | `Viper.Graphics.Camera.*` | 80+ LOC manual lerp camera in `camera.zia` |
| **SpriteSheet** | `Viper.Graphics.SpriteSheet.*` | Procedural pixel-by-pixel sprite construction (864 LOC) |
| **SpriteBatch** | `Viper.Graphics.SpriteBatch.*` | Individual `Canvas.BlitAlpha()` per entity per frame |
| **ScreenFX** (shake, flash, fade) | `Viper.Game.ScreenFX.*` | 50+ LOC manual shake/flash/fade in `renderer.zia` |
| **Scene/SceneNode** | `Viper.Graphics.Scene.*` | Manual draw-order management in `renderer.zia` |
| **Action mapping** | `Viper.Input.Action.*` | Raw `Keyboard.IsDown(keyCode)` checks |
| **Easing functions** | `Viper.Math.Easing.*` | Manual linear interpolation only |
| **Timer/Tween** | `Viper.Game.Timer/Tween` | Manual frame counters for everything |

**This is the single biggest issue.** The demo is ~3000 LOC when it could be ~1000 LOC using existing runtime APIs.

## WHAT THE DEMO DOES BY HAND (That the Runtime Already Provides)

### 1. Physics (~200 LOC → ~20 LOC)
**Current** (`physics.zia`): Manual gravity integration, x100 sub-pixel coordinates, separate X/Y axis collision sweep, bit-flag collision results unpacked with `(flags/4)%2`.

**Runtime has**: `Physics2D.World` with gravity, `Physics2D.Body` with velocity/impulse/collision layers, `Tilemap.CollideBody()` for tile collision resolution.

**Impact**: Entire `physics.zia` file could be replaced. Player/enemies become `Body` objects. Coyote time and jump buffering stay in game logic, but gravity/collision moves to runtime.

### 2. Particles (~150 LOC → ~30 LOC)
**Current** (`particles.zia`): 256-slot pool with parallel arrays (x, y, vx, vy, life, maxLife, size, color, active), manual gravity, fade calculation.

**Runtime has**: `ParticleEmitter` with configurable rate, lifetime range, velocity range, gravity, color, size, fade-out, shrink, burst mode.

**Impact**: Replace `dustPuff()`, `sparkBurst()`, `coinCollect()`, `deathExplosion()` etc. with emitter presets: `emitter.SetVelocity(-2,2,-3,0); emitter.SetLifetime(15,27); emitter.Burst(6)`.

### 3. Sprites & Animation (~900 LOC → ~100 LOC)
**Current** (`sprites.zia`): 864 LOC of procedural pixel art — every frame of every entity drawn with `Pixels.DrawBox()`, `Pixels.DrawDisc()`, `Pixels.SetRGB()`. Animation via manual `animTimer` counters with modulo frame selection. Sprite mirroring via `Clone() + FlipH()`.

**Runtime has**: `Sprite` with multi-frame animation, `FlipX`/`FlipY` properties, `SetFrameDelay()`, `Update()` auto-advances frames. `SpriteAnimation` provides named clips with loop/pingpong/speed control.

**Impact**: Still need procedural generation (no asset files), but sprites become `Sprite` objects with `AddFrame()`. Animation state machines use `SpriteAnimation`. FlipX replaces pre-cloning left/right variants.

### 4. Tilemap Rendering (~100 LOC → ~5 LOC)
**Current** (`renderer.zia` + `level.zia`): Manual viewport culling loop, per-tile type switch for color selection, `Canvas.Box()` per visible tile.

**Runtime has**: `Tilemap` with `SetTileset()` (SpriteSheet), `DrawRegion()` (auto-culled), `SetCollision()` per tile type, `IsSolidAt()`, `CollideBody()`.

**Impact**: Level builds a `Tilemap`, assigns collision flags. Rendering = `tilemap.DrawRegion(canvas, camX, camY, ...)`. Collision = `tilemap.CollideBody(playerBody)`.

### 5. Camera (~80 LOC → ~5 LOC)
**Current** (`camera.zia`): Manual lerp follow (1/8 X-speed, 1/12 Y-speed), bounds clamping, downward bias offset, world-to-screen coordinate conversion duplicated everywhere.

**Runtime has**: `Camera` with `Follow()`, `SetBounds()`, `ToScreenX/Y()`, `ToWorldX/Y()`, integrates with `Scene.DrawWithCamera()`.

**Impact**: Replace entire `camera.zia` with `camera.Follow(playerX, playerY)` + bounds. All rendering uses camera transforms automatically when using Scene.

### 6. Screen Effects (~50 LOC → ~5 LOC)
**Current** (`renderer.zia`): Manual shake offset via `(timer * 7919) % (intensity * 2) - intensity`, flash overlay via `BoxAlpha`, fade via alpha counter.

**Runtime has**: `ScreenFX.Shake(intensity, frequency, duration)`, `ScreenFX.Flash(color, duration)`, `ScreenFX.FadeIn/FadeOut(color, duration)`.

### 7. Input (~scattered → ~15 LOC setup)
**Current**: Raw `Keyboard.IsDown(75)` / `Keyboard.WasPressed(32)` (magic key codes) scattered across `player.zia`, `menu.zia`, `game.zia`.

**Runtime has**: `Action.Define("jump"); Action.BindKey("jump", Keys.SPACE); Action.BindKey("jump", Keys.W); Action.BindPadButton("jump", 0, PAD_A)`. Then `Action.Pressed("jump")` everywhere.

**Impact**: Enables gamepad support for free. Removes magic key codes. Centralizes input config.

## WHAT THE RUNTIME DOESN'T PROVIDE (Genuine Gaps)

These are features the sidescroller implements that have **no runtime equivalent**:

### Priority 1 — High Impact, Common Need

1. **Parallax Background Layers** — No API to define scrolling layers with different scroll factors. The camera exists but has no parallax support. Every scrolling game needs this.
   - **Proposal**: `Camera.AddParallaxLayer(pixels, scrollFactorX, scrollFactorY)` or a `ParallaxLayer` class

2. **Delta Time / Frame Timing** — No `Viper.Time.DeltaMs()` or frame-rate-independent timing. Demo uses `SleepMs(16)` for ~60fps but has no way to measure actual frame time. Physics is frame-locked.
   - **Proposal**: `Canvas.get_DeltaTime()` returning milliseconds since last `Flip()`, or `Viper.Time.Ticks()`

3. **ParticleEmitter.Draw()** — The emitter tracks particles but has **no Draw method** in runtime.def! Games must call `Get()` per particle and draw manually. This defeats most of the purpose.
   - **Proposal**: `ParticleEmitter.Draw(canvas)` or `ParticleEmitter.DrawAt(canvas, offsetX, offsetY)`

4. **Save/Load (Persistence)** — No file I/O for game state. High scores are memory-only, lost on exit.
   - **Proposal**: `Viper.IO.WriteText(path, data)` / `Viper.IO.ReadText(path)` at minimum

### Priority 2 — Medium Impact

5. **Sin/Cos/Atan2 Math** — No trig functions exposed. Bat hover uses `timer % 60 < 30` as sine approximation. Boss aiming has no angle calculation.
   - **Proposal**: `Viper.Math.Sin(degrees)`, `Cos`, `Atan2`, `Sqrt` — even fixed-point integer versions would help

6. **String Interpolation / Formatting** — Score display uses `Viper.Fmt.Int()` then concatenation. No printf-style formatting.
   - **Proposal**: `Viper.Fmt.Format("Score: {}", score)` or `"Score: ${score}"` syntax

7. **Random.Range(min, max)** — If not already present, this is used constantly for particle velocities, spawn positions, etc.

### Priority 3 — Nice to Have

8. **Collision Callbacks on Physics2D** — The physics world resolves collisions but has no way to notify game code *what* collided with *what*. Sidescroller needs stomp detection, hazard contact, pickup collection.
   - **Proposal**: `World.SetCollisionCallback(fn)` or `Body.get_Collisions()` list

9. **Sprite.DrawAt(canvas, x, y)** — Currently `Sprite.Draw(canvas)` uses the sprite's internal position. A `DrawAt` variant would let games position sprites in world space without mutating state.

10. **Music Looping with Intro** — Audio supports `Music.Play(loop)` but no intro-then-loop-body pattern common in games.

## RECOMMENDATIONS

### Phase 1: Fix the Demo (Use What Already Exists)
Rewrite the sidescroller to use the runtime's existing APIs. This is the highest-ROI change — zero runtime work needed, just demo refactoring:
1. Replace `physics.zia` with `Physics2D.World` + `Body` + `Tilemap.CollideBody()`
2. Replace `particles.zia` with `ParticleEmitter` (except Draw — see gap #3)
3. Replace `sprites.zia` sprite storage with `Sprite` objects + `SpriteAnimation`
4. Replace `camera.zia` with `Camera` + `SetBounds()`
5. Replace tile rendering with `Tilemap.DrawRegion()`
6. Replace screen effects with `ScreenFX`
7. Replace raw key codes with `Action` mappings (adds gamepad support)
8. Use `SpriteBatch` for entity rendering
9. Use `Timer`/`Tween`/`SmoothValue` (PAC-MAN already does this)

### Phase 2: Fill Runtime Gaps (Small Additions)
1. **Add `ParticleEmitter.Draw(canvas)`** — Without this, the emitter is awkward to use
2. **Add `Camera` parallax layer support** — Every 2D scroller needs this
3. **Add delta time API** — Frame-rate-independent physics is table stakes
4. **Expose trig functions** — `Sin`, `Cos`, `Atan2`, `Sqrt` at minimum

### Phase 3: Enhance the Demo (Make It Impressive)
With runtime APIs + new gaps filled, add polish:
- Smooth parallax scrolling backgrounds (3+ layers)
- Proper frame-rate-independent physics
- Gamepad support (already free with Action mapping)
- Rotating/scaling boss sprites using `Sprite.Rotation`
- Particle trails on projectiles
- Screen-space effects via ScreenFX
- Scene graph for proper draw ordering
- Saved high scores via file I/O

## CROSS-DEMO PATTERN: Same Problem in Every Game

The PAC-MAN, Centipede, and Chess demos also ignore these runtime APIs. A shared "game template" or updated documentation showing the idiomatic way to build a Viper game would prevent future demos from reinventing the wheel.
