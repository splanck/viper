## arrays/ (6 files) — AUDITED

### RT-001 — rt_array_obj.c:146-182 — Object leak on array shrink (P2)

`rt_arr_obj_resize` shrinks the logical length without releasing truncated elements. If an object array shrinks from len=10 to len=5, elements at indices 5-9 retain their reference counts. The comment on line 146 acknowledges this but the behavior leaks reference-counted objects.

**Recommendation:** Release elements in the truncated range before shrinking, or document that callers MUST clear elements before shrinking.

### RT-002 — rt_array_obj.c:167 — Raw realloc bypasses heap tracking (P2)

`rt_arr_obj_resize` calls `realloc(hdr, total_bytes)` directly on the heap header. If the heap allocator maintains metadata about allocations (GC roots, allocation tracking), this raw realloc bypasses that tracking and could cause GC to miss or double-scan the allocation.

**Recommendation:** Use the heap allocator's resize API instead of raw realloc, or verify that the heap allocator doesn't maintain separate tracking.

### RT-003 — rt_list_i64.c:82 — Integer overflow in capacity doubling (P1)

```c
size_t new_cap = cap ? cap * 2 : RT_LIST_I64_MIN_CAP;
```

No overflow check. If `cap > SIZE_MAX / 2`, the multiplication wraps to a small value. The subsequent `rt_heap_alloc` with the small capacity succeeds, but `memcpy(new_arr, arr, len * sizeof(int64_t))` on line 89 copies `len` elements into a buffer sized for `new_cap` (which wrapped to a small number), causing a heap buffer overflow.

**Recommendation:** Add: `if (cap > SIZE_MAX / (2 * sizeof(int64_t))) rt_trap("list capacity overflow");`

### RT-004 — rt_array.c:189, rt_array_f64.c:168, rt_array_i64.c:183 — Multiplication overflow in copy_payload (P3)

`count * sizeof(T)` has no overflow check in `copy_payload` functions. Practically bounded by heap header values, so exploitation requires a corrupted header.

**Recommendation:** Add defensive overflow check: `if (count > SIZE_MAX / sizeof(T)) rt_trap("copy overflow");`

### RT-005 — All array files — assert-only validation in release builds (P3)

`rt_arr_*_assert_header()` uses `assert()` which compiles to nothing in release builds. Corrupted headers silently proceed, potentially causing memory corruption.

**Recommendation:** Replace with runtime checks that call `rt_trap()` for safety-critical header validation.

## audio/ (5 files) — AUDITED

### RT-006 — rt_audio.c:158 — __atomic_test_and_set not available on MSVC (P1)

The spinlock at line 158 uses `__atomic_test_and_set` and line 176/190/224 uses `__atomic_clear`, but the MSVC compatibility guards (lines 150-153) only cover `__atomic_load_n`/`__atomic_store_n`. These atomic builtins are GCC/Clang-only and will fail to compile on MSVC.

**Recommendation:** Add MSVC equivalents using `_InterlockedExchange` for test-and-set and `_InterlockedExchange` with 0 for clear. Wrap in `#if RT_COMPILER_MSVC` guards.

### RT-007 — rt_audio.c:650-694 — Crossfade stores dangling music pointers (P1)

`rt_music_crossfade_to` stores raw `void*` pointers to music objects in `g_crossfade.fade_out`/`fade_in` without retaining them. If the caller releases the music object between starting and completing the crossfade, the stored pointers become dangling. Subsequent `rt_music_stop`/`rt_music_set_volume` calls in `rt_music_crossfade_update` are use-after-free.

**Recommendation:** Retain the music objects when starting a crossfade and release them when the crossfade completes or is cancelled.

### RT-008 — rt_audio.c:79 — Spinlock type mismatch (P2)

`g_audio_init_lock` is declared as `volatile int` but `__atomic_test_and_set` expects `volatile void*` or `volatile unsigned char`. The behavior is implementation-defined on some compilers.

**Recommendation:** Change type to `volatile unsigned char` or use `_Atomic int` with `atomic_exchange`.

### RT-009 — rt_audio_codec.c:238 — __attribute__((packed)) not portable to MSVC (P1)

The `vaf_header` struct uses `__attribute__((packed))` which is GCC/Clang-only. MSVC requires `#pragma pack(push, 1)` / `#pragma pack(pop)`.

**Recommendation:** Use a cross-platform packing macro, e.g.:
```c
#if defined(_MSC_VER)
#pragma pack(push, 1)
#endif
typedef struct { ... }
#if defined(__GNUC__) || defined(__clang__)
__attribute__((packed))
#endif
vaf_header;
#if defined(_MSC_VER)
#pragma pack(pop)
#endif
```

### RT-010 — rt_audio_codec.c:289 — Unbounded allocation from untrusted file header (P2)

`malloc((size_t)total * sizeof(int16_t))` where `total` comes directly from the `.vaf` file header without any upper bound check. A malicious file could set `total_samples` to trigger a multi-gigabyte allocation.

**Recommendation:** Add a maximum sample count check: `if (total > 48000 * 3600 * 2) return NULL;` (2 hours of 48kHz stereo audio ≈ 345M samples).

### RT-011 — rt_audio_codec.c:386 — Division by zero on malformed WAV (P2)

```c
result.sample_count = total_samples / result.channels;
```

If `channels == 0` in a malformed WAV file, this is division by zero (undefined behavior / crash).

**Recommendation:** Add: `if (result.channels == 0) { fclose(f); return result; }`

### RT-012 — rt_audio_codec.c:350-355 — WAV parser assumes 44-byte header (P3)

`parse_wav` reads exactly 44 bytes and assumes the data chunk starts immediately after the format chunk. WAV files with extra chunks (LIST, INFO, fact, etc.) between fmt and data will be incorrectly parsed.

**Recommendation:** Implement proper chunk scanning: read RIFF header, then iterate through chunks by ID and size until "data" is found.

### RT-013 — rt_playlist.c:186 — memset wipes object header (P1)

```c
playlist_impl *pl = (playlist_impl *)rt_obj_new_i64(0, sizeof(playlist_impl));
memset(pl, 0, sizeof(playlist_impl));
```

`rt_obj_new_i64` allocates and initializes the object with a header (refcount, type info). The immediately following `memset(pl, 0, ...)` zeroes the entire allocation including whatever header precedes the payload. This corrupts the refcount and type metadata.

**Recommendation:** Either (a) verify that `rt_obj_new_i64` returns a pointer past the header (so memset only hits the payload), or (b) remove the memset and rely on the field-by-field initialization that follows.

### RT-014 — rt_playlist.c:92-97 — Non-thread-safe srand/rand for shuffle (P2)

Uses `srand(time(NULL))` with a non-atomic `seeded` flag and `rand()` which is not thread-safe. Multiple threads calling playlist functions simultaneously could corrupt the global RNG state.

**Recommendation:** Use the runtime's thread-safe random functions (`rt_random_*`) or a local PRNG state.

### RT-015 — rt_playlist.c:101 — Modulo bias in Fisher-Yates shuffle (P3)

`rand() % (i + 1)` has modulo bias when `RAND_MAX + 1` is not divisible by `i + 1`. Produces slightly non-uniform distributions.

**Recommendation:** Use rejection sampling or the runtime's random API.

### RT-016 — rt_soundbank.c:281 — Borrowed reference returned without retain (P2)

`rt_soundbank_get` returns the stored sound pointer without retaining it. If the bank is modified (remove/clear) after the caller receives the pointer, the caller holds a dangling reference.

**Recommendation:** Retain before returning and document that the caller must release.

### RT-017 — rt_synth.c:303-306 — Comment says "exponential" but code is "quadratic" (P3)

Comment reads "Exponential decay envelope" but the code computes `env = (1.0 - t)^2` which is quadratic decay, not exponential.

**Recommendation:** Fix comment to "Quadratic decay envelope".

## collections/ (55 files) — IN PROGRESS

### RT-018 — rt_bag.c:9-14, 88-94 — File describes "multiset (Bag)" but implements Set (P2)

The file header and struct docstring say this is a "multiset (Bag) that tracks how many times each distinct string has been added." However, `rt_bag_entry` has no count field and `rt_bag_add` (line 391-394) rejects duplicates. This is a **Set**, not a Bag/multiset. The API documentation and all consumer documentation built from these comments is fundamentally wrong.

**Recommendation:** Either (a) rename to Set and update docs, or (b) add a `count` field to `rt_bag_entry` and implement true multiset semantics (increment on duplicate add, decrement on remove).

### RT-019 — rt_bag.c:709-711, 725-727, 795-798, 868-870, 891-894 — Temporary string leak in set operations (P2)

In `rt_bag_union`, `rt_bag_intersect`, `rt_bag_diff`, `rt_bag_clone`, and `rt_bag_items`: `rt_string_from_bytes` creates a new string (refcount=1) that is pushed into the result bag/seq. `rt_bag_add` copies the key data internally (via `malloc`+`memcpy`), so the temporary `rt_string` is never released. Each call leaks one string object.

**Recommendation:** Release the temporary string after `rt_bag_add`/`rt_seq_push`: `rt_str_release_maybe(str);`

### RT-020 — rt_bag.c:855 — Wrong docstring on rt_bag_clone (P3)

The docstring above `rt_bag_clone` is the docstring for `rt_bag_diff` (set difference documentation). The clone function has the wrong documentation.

### RT-021 — rt_bimap.c:248-253 — Object leak on partial init failure (P2)

If `fwd_buckets` allocation succeeds but `inv_chains` fails, the `bm` object (allocated by `rt_obj_new_i64`) is leaked — we return NULL without releasing it. No finalizer was set, so the GC won't collect it.

**Recommendation:** Set the finalizer before the calloc calls, or call `rt_obj_free(bm)` in the error path.

### RT-022 — rt_bimap.c:64-65 — Dead field `inv_buckets` (P3)

`inv_buckets` is declared in the struct but never used. The inverse lookup actually uses `inv_chains` (line 75). Dead code that wastes 8 bytes per BiMap and misleads readers.

### RT-023 — rt_bimap.c:470-471, 488-489 — Temporary string leak in keys/values (P2)

Same pattern as RT-019. `rt_string_from_bytes` creates strings that are pushed to the seq but never released, leaking one string per entry.

### RT-024 — rt_binbuf.c:77 — Integer overflow in position+needed (P1)

`int64_t required = buf->position + needed` — if both values are large positive int64, the sum overflows to negative, making the `required <= buf->capacity` check pass. Subsequent write goes past buffer end.

**Recommendation:** `if (needed > INT64_MAX - buf->position) rt_trap("BinaryBuffer: position overflow");`

### RT-025 — rt_binbuf.c:131 — Same overflow in check_read (P2)

### RT-026 — rt_binbuf.c:460-461 — Byte-by-byte read_bytes vs memcpy (P3)

### RT-027 — rt_bitset.c:87, 91 — Grow overflow: word_count*2 and realloc size (P2)

`bs->word_count * 2` has no overflow check, and `grow * sizeof(uint64_t)` in the realloc can also overflow SIZE_MAX.

## Systemic Patterns Identified

### Pattern S1 — `memset` after `rt_obj_new_i64` wipes object header (P1, 4+ instances)

Files: rt_playlist.c:186, rt_pathfollow.c:144, rt_quadtree.c:446, rt_screenfx.c:89

`rt_obj_new_i64` allocates and initializes the object with header metadata (refcount, type info). The immediately following `memset(obj, 0, sizeof(...))` zeroes the entire allocation including the header. This corrupts the refcount and type metadata.

**Fix:** Either verify `rt_obj_new_i64` returns a pointer past the header, or remove the memset and initialize fields individually.

### Pattern S2 — `rt_string_from_bytes` temporary leak in hash table `keys()`/`items()` (P2, 12+ instances)

Files: rt_bag, rt_bimap, rt_countmap, rt_defaultmap, rt_frozenset, rt_lrucache, rt_map, rt_multimap, rt_orderedmap, rt_treemap, rt_trie, rt_pathfinder

`rt_string_from_bytes` creates a new string (refcount=1) that is pushed into a Seq via `rt_seq_push`. Since Seq by default does NOT retain elements (stores raw pointers), the string's refcount stays at 1 and is the only reference. But the string is never released by the caller, so it leaks.

**Fix:** After `rt_seq_push(seq, str)`, the string should be released if the seq doesn't retain it. Or use `rt_seq_set_owns_elements(seq, 1)` before pushing.

### Pattern S3 — Missing SIZE_MAX overflow check on `malloc(cap * sizeof(T))` (P2, 8+ instances)

Files: rt_deque, rt_queue, rt_pqueue, rt_bitset, rt_list, rt_stack, rt_seq, rt_ring

When capacity grows, the `new_cap * sizeof(T)` product is passed to malloc/realloc without checking for SIZE_MAX overflow. If `new_cap` is near SIZE_MAX/sizeof(T), the product wraps to a small value, causing a heap buffer overflow.

**Fix:** Add `if ((size_t)new_cap > SIZE_MAX / sizeof(T)) rt_trap("...: size overflow");` before allocation.

### Pattern S4 — Object leak on partial init failure (P2, 7+ instances)

Files: rt_bimap, rt_countmap, rt_objpool, rt_pathfinder, rt_quadtree, rt_grid2d, rt_unionfind

When `rt_obj_new_i64` succeeds but a subsequent allocation fails, the constructor returns NULL without releasing the initial object. No finalizer is set yet, so the GC can't collect it.

**Fix:** Set the finalizer before the secondary allocations, so the finalizer can clean up even if the object is only partially initialized.

### Pattern S5 — `rt_obj_free` without `rt_obj_release_check0` (P2, 7+ instances)

Files: rt_collision, rt_pathfollow, rt_screenfx, rt_smoothvalue, rt_timer, rt_tween, rt_dialogue(set_font)

`destroy` functions call `rt_obj_free` directly. If the object has refcount > 1 (other references exist), this is use-after-free for the other holders.

**Fix:** Replace `rt_obj_free(obj)` with `if (rt_obj_release_check0(obj)) rt_obj_free(obj);`

### Pattern S6 — Comment says "values not retained" but code retains them (P3, 6+ instances)

Files: rt_frozenmap, rt_orderedmap, rt_defaultmap, rt_treemap, rt_set, rt_countmap

File headers claim values are stored as raw pointers, but the implementation retains them on insert and releases on remove. The comments mislead readers and could cause incorrect caller behavior.

**Fix:** Update comments to accurately describe the retain/release semantics.

Comment says "Cannot reset range without original start value." After consuming a range, `reset()` doesn't restore the starting value, so subsequent iteration produces nothing.

### RT-262 — rt_lazyseq.c:600-601 — `reset` is a no-op for counted repeat sequences (P2)

Same issue: "Cannot reset count." After consuming `repeat(val, 5)`, reset doesn't restore the count.

### RT-263 — rt_lazyseq.c:335 — No integer overflow check on range step addition (P3)

`seq->data.range.current = cur + step` can overflow for values near INT64_MAX/MIN.

### RT-264 — rt_lazyseq.c:957-961 — `w_next` wrapper discards `has_more` flag (P3)

IL callers cannot distinguish end-of-sequence from a legitimate NULL element.

### RT-265 — rt_result.c:77,86,95,104,113,122 — No NULL check after rt_obj_new_i64 (P2)

All 6 creation functions dereference immediately. Same as RT-255.

### RT-266 — rt_result.c (all creation functions) — No GC finalizer set (P2)

Held ptr/str references never released when Result is freed — memory leak.

### RT-267 — rt_result.c (all creation functions) — Don't retain passed values (P2)

`rt_result_ok(val)` stores `val` without retaining. Dangling reference if caller releases.

### RT-268 — rt_result.c:449 — snprintf buf[256] truncates long string values (P3)

### RT-269 — rt_result.c:428 — `rt_result_equals` f64 uses `==`, NaN != NaN (P3)

### RT-270 — rt_type_registry.c:503,512 — Dangling class name from `rt_string_cstr` (P2)

`rt_register_class_direct_rs` and `rt_register_class_with_base_rs` store the raw pointer from `rt_string_cstr()` into the registry. If the runtime string is freed, the class name is a dangling pointer. Should `strdup`.

### RT-271 — rt_type_registry.c:201-222 — O(n) linear scans for class lookup (P3)

`find_class_by_type`/`find_class_by_vptr` are O(n) linear scans. O(d*n) total cost for inheritance checks. Consider hash map for larger registries.

### RT-272 — rt_type_registry.c:334-346 — No cycle detection in `rt_type_is_a` (P3)

Walks the inheritance chain via `find_class_by_type`. Circular inheritance (bug in code generator) causes infinite loop.

### RT-273 — rt_type_registry.c:all registration functions — No thread safety (P3)

Header says "process-global singleton; all threads share it" but no locking on registration operations. Concurrent registration corrupts shared arrays.

## text/ (30 files)

### RT-277 — rt_aes.c:876-879 — Memory leak in `generate_iv` (P2)

`rt_crypto_rand_bytes(16)` returns a Bytes object that is never freed. Leaks memory on every `rt_aes_encrypt_str` call.

### RT-278 — rt_aes.c:816 — Padding oracle via distinct error message (P2 SECURITY)

`rt_aes_decrypt` traps with "invalid padding or corrupted data." While PKCS7 check is constant-time (S-05), the distinct error can enable padding oracle attacks if observable.

### RT-279 — rt_aes.c:921-926,956-966 — Per-byte Bytes access in encrypt/decrypt (P3)

Uses `rt_bytes_get`/`rt_bytes_set` loops instead of memcpy. Inefficient.

### RT-280 — rt_aes.c:840-855 — Fixed salt in key derivation (P3)

`derive_key` uses a hardcoded salt. Same password always yields same key. Comments acknowledge.

### RT-281 — rt_cipher.c:100,155,212 — `memset(key,0)` may be optimized out (P2 SECURITY)

Compiler may elide stack variable zeroing for locals going out of scope. Use `memset_explicit()` (C23) or volatile write to ensure cryptographic key scrubbing.

### RT-282 — rt_cipher.c:60-64 — Fragile Bytes layout assumption (P3)

`bytes_impl` struct assumes specific internal layout of Bytes objects. Will break silently if layout changes.

### RT-283 — rt_compiled_pattern.c:70-81 — No finalizer for compiled regex (P2)

`re_compiled_pattern` is never freed when the wrapper object is collected by GC.

### RT-284 — rt_compiled_pattern.c:79 — No NULL check on `re_compile` return (P3)

### RT-285 — rt_csv.c:54 — Type confusion in box detection (P3)

`csv_extract_string` reads first 8 bytes as int64_t tag. Non-boxed memory coincidentally matching `RT_BOX_STR` causes incorrect unboxing.

### RT-286 — rt_diff.c:140 — No NULL check on calloc for LCS table (P2)

Crash on large inputs when allocation fails.

### RT-287 — rt_diff.c:101-118 — O(nm) space LCS table (P3)

Two 10K-line files require ~400MB. Hirschberg's algorithm would use O(n).

### RT-288 — rt_html.c:521 — Numeric entities only for ASCII (P3)

`code < 128` check in unescape means Unicode entities above U+007F are silently ignored.

### RT-289 — rt_html.c:212 — Fixed 256-depth stack in HTML parser (P3)

Extreme nesting silently truncated. Sufficient for real HTML.

### RT-290 — rt_hash.c:488-499 — SHA-256 processes one byte at a time (P2)

`sha256_update` uses per-byte loop instead of block buffering like MD5/SHA-1. SHA-256 is 10-100x slower than it should be for large inputs.

### RT-291 — rt_hash.c:1163 — Comment says "SipHash-2-4" but calls `rt_fnv1a` (P3)

Comment-implementation mismatch.

### RT-292 — rt_hash.c:19 — CRC32 table init not thread-safe (P3)

Header acknowledges: "callers must externally synchronize if called concurrently before init."

### RT-293 — rt_rand.c:133-136 — Per-byte `rt_bytes_set` loop (P3)

Inefficient for large random byte generation. Should use raw pointer access.

### RT-294 — rt_password.c:263 — No NULL check on `hash` parameter (P2)

`rt_string_cstr(NULL)` may return NULL, causing `strncmp(NULL, ...)` UB.

### RT-295 — rt_regex.c (backtracking engine) — ReDoS vulnerability (P2 SECURITY)

Backtracking NFA with no catastrophic backtracking protection. Patterns like `(a+)+b` cause exponential matching time against crafted input.

### RT-296 — rt_regex.c:24 — Pattern cache entries never freed (P2)

"entries are not freed" — unbounded memory growth for long-running applications using many unique patterns.

### RT-297 — rt_json.c:215-293 — Unicode \u handler missing surrogate pairs (P3)

`\uXXXX` handler doesn't combine high+low surrogates (\uD800-\uDBFF + \uDC00-\uDFFF). Characters above U+FFFF silently mis-encoded.

### RT-298 — rt_serialize.c:53 — Thread-local error string leaked (P2)

`_Thread_local rt_string g_last_error` allocated via `rt_string_from_bytes` but previous allocation never freed before replacement.

## threads/ (13 files)

### RT-299 — rt_channel.c:626-641 — Property reads without monitor (P3)

`rt_channel_get_cap` and `rt_channel_get_is_closed` read shared fields without acquiring the monitor. Safe on x86 (TSO), potentially stale on ARM.

### RT-300 — rt_channel.c:279-356,495-579 — Timed send/recv without absolute deadlines (P3)

Uses relative timeouts in wait loops. Spurious wakeups can cause total wait time to exceed the specified timeout.

### RT-301 — rt_concqueue.c:23 vs 164 — Header says "not retained" but code retains (P3)

Comment claims "Enqueued void* values are not retained" but `rt_concqueue_enqueue` at line 164 calls `rt_obj_retain_maybe(item)`.

### RT-302 — rt_concqueue.c:203-222 — Blocking dequeue with no finalization escape (P3)

`rt_concqueue_dequeue` blocks indefinitely with `CQ_WAIT`. No mechanism to unblock if the queue is being finalized by GC while a thread is waiting.

### RT-303 — rt_future.c (all) — No finalizer on promise_impl (P2)

`pthread_mutex_t` and `pthread_cond_t` (or Windows CRITICAL_SECTION) in promise_impl are never destroyed when the object is GC'd. OS resource leak per promise.

### RT-304 — rt_future.c:164 — Promise stores value without retain (P2)

`rt_promise_set` does `p->value = value` without `rt_obj_retain_maybe`. If the caller releases the value after resolving, the promise holds a dangling reference.

### RT-305 — rt_future.c:126 — Future holds raw pointer to promise (P2)

No mutual retain between promise and future. If the promise is GC'd before the future, the future's `promise` pointer is dangling.
