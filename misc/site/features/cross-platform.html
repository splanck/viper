Cross-Platform

Every feature in Viper works on macOS, Linux, and Windows with full parity. This isn't "it compiles on Linux" — it means platform-native graphics, audio, networking, and threading on all three operating systems, both ARM64 and x86-64.

The Abstraction Layer

Platform differences are concentrated into two shared headers: src/runtime/rt_platform.h (runtime layer) and src/common/PlatformCapabilities.hpp (compiler layer). Together they provide:

  • Host detection — VIPER_HOST_WINDOWS / VIPER_HOST_MACOS / VIPER_HOST_LINUX
  • Compiler detection — VIPER_COMPILER_MSVC / VIPER_COMPILER_CLANG / VIPER_COMPILER_GCC
  • Capability macros — VIPER_CAN_FORK, VIPER_HAS_X11, VIPER_HAS_ALSA, VIPER_NATIVE_LINK_X86_64, etc.
  • Portable atomic operations — 32-bit, 64-bit, pointer, and size_t variants (32-bit `__atomic_compare_exchange_n` is MSVC-safe; 64-bit uses direct `_InterlockedCompareExchange64` on Windows)
  • Memory barrier abstractions — DMB on ARM64, MFENCE on x86, compiler barriers
  • Thread-local storage — portable TLS macros
  • High-resolution timing — clock_gettime() on POSIX, QueryPerformanceCounter on Windows
  • POSIX compatibility shims — Windows remappings for lseek64, strtok_r, path separators

The rest of the runtime compiles identically on all platforms, using only these portable abstractions. A `lint_platform_policy.sh` lint script enforces that raw `_WIN32`/`__APPLE__`/`__linux__` checks only appear in approved adapter files.

All build scripts auto-detect the OS, available compilers, and core count for parallel builds. CMake 3.20+ with C++20 support is the only requirement.

Native Code Generation

The compiler generates native executables for all 6 platform+architecture combinations:

  • ELF for Linux (x86-64 + ARM64)
  • Mach-O for macOS (x86-64 + ARM64 with 16KB page alignment)
  • PE for Windows (x86-64 + ARM64)

No external assembler, linker, or toolchain needed. See 6 Native Targets for details.
