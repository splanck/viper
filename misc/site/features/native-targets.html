6 Native Targets

Viper compiles directly to native executables without calling any external assembler or linker. The entire pipeline — from IL to relocatable object files to linked executables — is built from scratch.

macOS on ARM64 requires 16 KB page alignment — Apple's dynamic linker (dyld) rejects executables with incorrect alignment. Viper handles this correctly.

Compilation Pipeline

The native compilation pipeline has 6 stages:

  1. IL → Machine IR — Lower typed IL instructions to target-specific machine IR opcodes
  2. Legalization — Normalize instructions to legal forms for the target architecture
  3. Register Allocation — Linear-scan allocation with coalescing and spilling
  4. Peephole Optimization — 5-7 backend passes (strength reduction, copy propagation, DCE, branch optimization)
  5. Binary Encoding — MIR → machine code bytes + relocations
  6. Linking — Object files + runtime archives → native executable

Custom Binary Encoders

x86-64 Encoder (1,451 LOC)

Variable-length instruction encoding (1-15 bytes) handling:

  • REX prefix generation (W/R/X/B bits for 64-bit mode and extended registers)
  • ModR/M and SIB byte construction for all addressing modes
  • Displacement and immediate field packing
  • RIP-relative addressing with -4 addend correction
  • SSE2 scalar double operations (FADD, FSUB, FMUL, FDIV)

AArch64 Encoder (1,883 LOC)

Fixed-width 32-bit instructions via pure bit-field construction:

  • 3-register arithmetic with optional shifts
  • Load/store pairs (LDP/STP) for efficient prologue/epilogue
  • ADRP + ADD page-relative addressing for cross-section references
  • Branch26 relocations for function calls
  • Conditional branch encoding (B.cond, CBZ, CBNZ)

Custom Native Linker (8,162 LOC)

The linker pipeline handles the full journey from .o files to executables:

  • Object file reader — Auto-detects ELF, Mach-O, or COFF by magic bytes
  • Archive reader — Parses GNU, BSD, and COFF archive variants (.a / .lib)
  • Symbol resolver — Iterative fixed-point demand-pull with Global > Weak precedence
  • Section merger — Classifies and concatenates .text, .rodata, .data, .bss with alignment
  • Relocation applier — Format-dispatched patching (PCRel32, Abs64, Branch26, Page21/PageOff12)
  • Dead code stripping — Eliminates unreferenced sections
  • ICF — Identical Code Folding to reduce binary size
  • Code signing — Required for macOS Mach-O executables

Register Allocation

Both architectures use linear-scan register allocation with additional optimization passes:

AArch64

  • Block-order traversal with cross-block persistence via exit-state caching
  • Copy coalescer integrated before linear scan (~270 LOC)
  • Next-use distance heuristics for spill victim selection
  • Protected-use eviction so the source operands of the currently-allocating instruction can never be evicted
  • `isLiveOut()` query prevents premature release for vregs with no remaining uses in the current block but live across CFG boundaries
  • Block-epoch spill slot lifetimes prevent stale offset reuse across blocks
  • Callee-saved register tracking for automatic prologue/epilogue generation
  • Apple M-series scheduler latency tuning (FP divide 10 cycles, integer divide 7, FP multiply 4)

x86-64

  • Pre-computed live intervals for priority-ordered assignment
  • Aggressive copy coalescing as post-allocation pass
  • Dedicated spiller with reload/store pair insertion
  • Carry-through across simple block boundaries (`canCarryIntoNextBlock()`) avoids forced spills at single-predecessor fallthrough edges
  • Branch relaxation: short JMP (2 bytes) and short Jcc (2 bytes) for near branches replace always-near 5-byte / 6-byte forms
  • 16 GPR + 16 XMM register tracking

Backend Peephole Optimization

After register allocation, both backends apply target-specific peephole passes:

AArch64 (7+ passes)

  • BranchOpt — Jump-to-jump elimination, unreachable block removal
  • CopyPropDCE — Forward copy propagation + dead code elimination
  • IdentityElim — Self-move removal (MOV Xn, Xn)
  • LoopOpt — Loop-invariant constant hoisting; multi-block phi-spill rewriting
  • MemoryOpt — Redundant load/store elimination
  • StrengthReduce — Multiply/divide by power-of-2 → shift/AND
  • JoinPhiCoalesce — Eliminates stack round-trips at CFG join blocks by proving all predecessors store the same physical registers and replacing the load prefix with topologically-ordered moves
  • Scheduler / BlockLayout — PassManager-driven post-RA scheduling and block layout at O1+

x86-64 (5 passes)

  • ArithSimplify — Zero-extension elimination, commutative reordering
  • BranchOpt — Jump optimization and unreachable detection
  • DCE — Dead instruction removal with fixed-point iteration
  • MovFolding — MOV constant propagation, multiply-by-power-of-2 → shift

Both backends share template-based peephole infrastructure for DCE and copy propagation, reducing code duplication.

Security Hardening

PAC/BTI on AArch64

Viper emits ARM Pointer Authentication and Branch Target Identification instructions:

  • PACIASP (HINT #25) — Signs the link register with the stack pointer at function entry, preventing ROP attacks
  • AUTIASP (HINT #29) — Verifies the link register before return
  • BTI C (HINT #34) — Marks valid indirect call targets, preventing jump-oriented attacks

These are encoded as HINT instructions that execute as NOPs on pre-ARMv8.3 hardware, so security is on by default with zero performance cost on older chips.

Stack Protection

  • ELF: PT_GNU_STACK with non-executable flag
  • PE: NX_COMPAT and HIGH_ENTROPY_VA DLL characteristics
  • Mach-O: PIE (Position-Independent Executable) by default
