Zero Dependencies

Viper has no external dependencies. No LLVM, no Boost, no SDL, no OpenSSL, no libcurl, no nlohmann/json, no Qt, no zlib. Every component of the compiler, runtime, and toolchain is written from scratch in C++ and C.

This isn't a philosophical choice about purity — it's a practical decision that gives Viper complete control over its compilation pipeline, enables portability to unusual targets (like ViperDOS), and makes the entire system auditable and educational.

Custom Binary Encoders

Viper doesn't call out to as, nasm, or LLVM to generate machine code. It has its own binary encoders that translate Machine IR directly into raw bytes:

  • x86-64 Encoder (1,451 LOC) — REX prefixes, ModR/M/SIB byte construction, displacement/immediate packing, RIP-relative addressing
  • AArch64 Encoder (1,883 LOC) — Fixed-width 32-bit instruction encoding, ADRP+ADD page-relative pairs, STP/LDP prologue synthesis

These encoders handle all the gnarly details: register encoding, addressing mode selection, immediate field sizing, and relocation emission.

Custom Native Linker

The native linker is 8,162 lines of hand-written code that takes object files and produces executables. No ld, no lld, no external toolchain:

  • Archive reader — Parses GNU, BSD, and COFF archive formats (.a / .lib)
  • Symbol resolver — Iterative demand-pull resolution with Global > Weak > Undefined precedence
  • Section merger — Classifies and concatenates .text, .rodata, .data, .bss with proper alignment
  • Relocation applier — Format-dispatched patching: PCRel32, Abs64, Branch26, Page21/PageOff12
  • Dead code stripping — Eliminates unreferenced sections
  • ICF — Identical Code Folding to reduce binary size

Three executable writers produce platform-native binaries:

Custom Object File Writers

Before linking, the compiler writes relocatable object files in all three major formats (3,618 LOC total):

  • ELF writer — EM_X86_64 and EM_AARCH64, .rela sections with explicit addends
  • Mach-O writer — 0xFEEDFACF magic, underscore-prefixed symbols, compact 8-byte relocations, LC_BUILD_VERSION
  • COFF writer — IMAGE_FILE_MACHINE_AMD64/ARM64, 10-byte relocations with implicit addends

Custom Data Structures

Even foundational utilities are built in-house rather than pulling in Boost or similar:

  • SmallVector — Stack-optimized vector with inline storage for small sizes
  • Arena — Bump-allocated arena with tracked destructor support
  • StringInterner — String deduplication pool for deterministic symbol names
  • DiagExpected — Result/Expected type for error propagation
  • SourceManager — Source file tracking with line/column mapping
  • Diagnostics — Full diagnostic engine with error codes and source locations

Why It Matters

Complete Control

When a bug appears in the linker, the optimizer, or the runtime — we fix it directly. No waiting for upstream releases, no version conflicts, no ABI compatibility concerns.

Portability

Viper can target platforms where external libraries don't exist. ViperDOS runs on bare-metal x86 with no OS — something impossible if the compiler depended on LLVM or SDL.

Auditability

Every line of code in Viper's output binary was written by the Viper project. No hidden transitive dependencies, no supply chain concerns for the compiler itself.

Educational Value

Students and hobbyists can read the complete source of a working assembler, linker, optimizer, and runtime. There's no black box.
