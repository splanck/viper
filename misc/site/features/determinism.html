VM/Native Determinism

Every Viper program produces identical output whether run on the IL interpreter or compiled to a native ARM64/x86-64 binary. This is a core design invariant, not an aspiration.

Why It Matters

  • Fast iteration — Develop and test on the VM (instant startup, no compilation wait), then compile to native for production
  • Confidence — If it works on the VM, it works as a native binary. No "it only breaks when compiled" surprises
  • Debugging — Reproduce any native bug on the VM with full tracing (--trace, --trace-calls)

How It's Enforced

Deterministic IL Semantics

The Viper IL specification (v0.2.0) defines precise semantics for every opcode. Both the VM and native backends implement the same specification:

  • Signed/unsigned integer operations with defined overflow behavior
  • IEEE 754 double-precision floating-point
  • Deterministic symbol interning and label assignment
  • Identical runtime function ABI for both execution modes

Differential Testing

The test suite includes property-based differential tests that generate random-but-valid IL programs and verify that the VM and native backend produce identical output:

  • Random IL program generation with configurable parameters
  • Seeded RNG for reproducibility (configurable via VIPER_DIFF_ITERATIONS)
  • Checked arithmetic operations (iadd.ovf, isub.ovf, imul.ovf, sdiv.chk0)
  • Automatic comparison of VM output vs. native output

Dispatch Strategy Equivalence

The VM itself supports three dispatch strategies, and all three are verified to produce identical results:

  • Function table dispatch — Portable, default on all compilers
  • Switch dispatch — Inline switch statement
  • Threaded dispatch — Computed goto (GCC/Clang, fastest)

E2E Pipeline Tests

End-to-end tests verify the complete pipeline from source through IL through execution:

  • Frontend compilation equivalence (same source → same IL)
  • Optimizer transformation stability (optimized IL → same output)
  • Code generation correctness (native binary → same output as VM)
  • Demo execution verification (real programs produce expected output)

VM Architecture

The VM is a register-file interpreter with SSA value tracking:

  • ~100 IL opcodes covering arithmetic, memory, control flow, type conversion, exception handling, and traps
  • Frame management with locals, evaluation stack, and block state
  • Value representation via tagged Slot (int64, f64, ptr, str)
  • Call stack with frame buffer pooling (8 register files, 16 stack buffers)
  • Structured exception handling with error types and recovery points
  • Tail-call optimization (gated by VIPER_VM_TAILCALL)
  • Execution tracing for debugging (--trace, --trace-calls)

The Workflow

  1. Develop — Write code in Zia or BASIC, run instantly on the VM
  2. Test — Full debugging with tracing, breakpoints, and immediate feedback
  3. Ship — Compile to native for production deployment, same behavior guaranteed
