# ADR 0121: Give Network Policies Stable Identity and Atomic Retry State

## Status

Accepted

## Context

`RetryPolicy` and `RateLimiter` were stateful managed objects allocated with
class id zero. Their methods accepted an untyped object and immediately cast
its payload, so a wrong-class or undersized managed object could be read and
mutated as policy state. Constructors also dereferenced a null allocation when
an embedder's trap hook returned.

Retry attempt accounting and its xorshift state were ordinary mutable fields.
Concurrent workers sharing one policy raced, lost attempt increments, and could
consume more or fewer than the configured retry count. Jitter was generated by
calling the operating-system CSPRNG for every exponential-policy construction,
although scheduling jitter has no cryptographic requirement. Near
`INT64_MAX`, the implementation added jitter before checking the maximum;
signed overflow could occur before the later clamp.

RateLimiter maintained a second platform-clock implementation with a racy
Windows frequency cache and an adjustable realtime fallback. The runtime
already has a cross-platform monotonic microsecond clock with synchronized
fallback behavior. Positive infinity was also accepted as a refill rate,
creating permanently non-finite arithmetic.

Stable managed class tags and validation alter the runtime C ABI contract, so
this decision is recorded explicitly.

## Decision

- Assign stable `RT_RETRY_CLASS_ID` and `RT_RATELIMIT_CLASS_ID` tags. Validate
  class identity and minimum payload size at every non-null public receiver
  boundary, and stop immediately after a returning trap.
- Constructors guard failed managed allocations before initializing payload
  fields.
- Reserve RetryPolicy attempts with an atomic compare-and-swap loop. Each
  successful `NextDelay` receives one distinct zero-based attempt; the counter
  never advances beyond `max_retries`. Read-only properties use atomic snapshot
  loads and `Reset` publishes zero atomically.
- Replace mutable xorshift state with an immutable per-policy seed and a
  deterministic SplitMix64-derived value for each attempt. Seeds combine a
  wrapping atomic construction sequence, monotonic time, and configuration.
  This supplies independent non-security jitter without consuming OS entropy.
- Bound additive jitter by both 25 percent of the base delay and the remaining
  `max_delay_ms - delay` headroom before addition. The addition therefore
  cannot overflow and never needs a post-overflow clamp.
- Use `rt_clock_ticks_us` for RateLimiter refill timestamps. Store timestamps as
  signed microseconds and convert only the elapsed interval to floating point.
  Non-advancing or failed clock samples leave bucket state unchanged.
- Normalize non-finite, zero, and negative refill rates to `1.0`. Saturate a
  non-finite or signed-unrepresentable accumulated credit to full capacity
  before converting it to a whole-token integer.
- RetryPolicy supports concurrent `NextDelay`, property, and Reset calls.
  RateLimiter retains its documented external-serialization requirement because
  refill and multi-field token updates must remain one transaction.

## Consequences

- Concurrent retry workers collectively consume exactly the configured number
  of attempts without data races in policy state.
- Retry jitter construction avoids an OS entropy syscall and cannot make an
  otherwise non-security policy depend on CSPRNG availability.
- Reset restarts the attempt-indexed jitter sequence for that policy seed.
- Exponential delays remain within `[0, max_delay_ms]`, including configurations
  within a few units of `INT64_MAX`.
- RateLimiter follows the same clock failure, monotonicity, and platform policy
  as other runtime timing facilities; its source no longer contains platform
  conditionals.
- Wrong-class and undersized handles trap instead of accessing unrelated object
  payloads.

## Alternatives Considered

- Add a native mutex to RetryPolicy: rejected because attempt reservation and
  per-attempt jitter can be made lock-free without another resource lifetime.
- Keep CSPRNG jitter: rejected because unpredictability is unnecessary and the
  syscall/entropy-failure cost is avoidable.
- Clamp after signed addition: rejected because overflowing signed arithmetic is
  undefined before the clamp executes.
- Make RateLimiter lock-free in the same change: rejected because atomically
  coordinating timestamps, integer balance, and fractional carry needs a
  broader representation change. Its external synchronization contract remains
  explicit.
